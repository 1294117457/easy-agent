### 问题 2: System Prompt 未生效 ❌

#### 问题描述

用户期望：
- 默认 Prompt 为空（无需修改）
- 选择自定义 Prompt 后，AI 应该遵循 Prompt 中的指令
- 当前问题：选择了自定义 Prompt，问"你可以做什么"时，AI 没有体现 Prompt 功能

#### 问题分析

```
用户操作：
1. 创建 Prompt（如："你是猫娘小美，说话要可爱"）
2. 激活该 Prompt
3. 问："你可以做什么"
4. AI 回答：通用回答，没有体现"猫娘"角色
```

**可能原因排查**：

| 排查项 | 检查方式 | 可能原因 |
|-------|---------|---------|
| A. Prompt 未激活 | 检查 `is_active=1` | 创建 Prompt 后未正确激活 |
| B. Prompt 未发送到 LLM | 检查日志 `[AgentService] Prompt 状态` | 代码路径问题 |
| C. Prompt 内容未生效 | 检查发送给 LLM 的消息 | 模型理解问题 |
| D. Prompt 被覆盖 | 检查 systemPrompt 字段 | 空字符串覆盖了实际内容 |

### 问题 3: 消息重复添加到历史 ❌

#### 问题描述

发送给 LLM 的消息列表中可能包含重复的用户消息，导致 AI 重复回答。

#### 当前代码分析

```typescript
// electron/core/application/AgentService.ts
const history = this.storagePort.getMessages(conversationId);
// history = [msg1_user, msg1_assistant, msg2_user, msg2_assistant]
// ↑ 最后一条可能是当前用户消息（如果前端已经先存了）

const historyWithoutCurrent = history.slice(0, -1);
// 问题：如果最后一条是当前消息，这里排除了它

const recentHistory = historyWithoutCurrent.slice(-10);
const langchainMessages = recentHistory.map(/* ... */);

// ⚠️ 然后又 push 了 userInput
langchainMessages.push(new HumanMessage(userInput));
// 导致：如果 history 最后一条 == userInput，则重复了
```

#### 问题分析图

```
场景：前端先调用 appendMessage 存入用户消息

后端 history = [msg1_user, msg1_assistant, msg2_user(当前)]
                                                              ↓
historyWithoutCurrent = [msg1_user, msg1_assistant, msg2_user]
                                                              ↓
recentHistory = [msg1_user, msg1_assistant, msg2_user]
                                                              ↓
langchainMessages.push(userInput) → [msg1_user, msg1_assistant, msg2_user, msg2_user]
                                                              ↓
❌ 消息重复！
```

#### 修改方案

**判断是否重复，避免重复添加**

```typescript
// electron/core/application/AgentService.ts
async sendMessage(conversationId, userInput, callbacks) {
  const history = this.storagePort.getMessages(conversationId);

  // ✅ 新增：检查最后一条是否等于当前用户消息
  const lastMessage = history[history.length - 1];
  const isLastMessageDuplicate =
    lastMessage?.role === 'user' &&
    lastMessage?.content === userInput;

  console.log('[AgentService] 历史消息数:', history.length);
  console.log('[AgentService] 最后一条消息:', lastMessage?.role, lastMessage?.content?.substring(0, 30));
  console.log('[AgentService] 是否重复:', isLastMessageDuplicate);

  // 排除重复的消息（只取前面的历史）
  const historyToUse = isLastMessageDuplicate
    ? history.slice(0, -1)
    : history;

  const recentHistory = historyToUse.slice(-10);

  // 构建 langchain 消息
  const langchainMessages: BaseMessage[] = recentHistory.map((m) =>
    m.role === 'user'
      ? new HumanMessage(m.content)
      : new AIMessage(m.content)
  );

  // ✅ 始终添加当前用户消息
  langchainMessages.push(new HumanMessage(userInput));

  // 构建最终消息列表
  const allMessages: BaseMessage[] = [
    ...(systemPrompt ? [new SystemMessage(systemPrompt)] : []),
    ...langchainMessages,
  ];

  console.log('[AgentService] 发送给 LLM 的消息数:', allMessages.length);
  console.log('[AgentService] 消息列表:', JSON.stringify(allMessages.map(m => ({
    role: m.getType(),
    content: (m.content as string).substring(0, 50)
  }))));

  // ... 后续逻辑
}
```

### 问题 4: 前后端状态不同步 ⚠️

#### 问题描述

前端维护独立的消息状态 `messages.value`，可能与后端数据库不一致。

#### 用户期望

1. **以后端为准**：消息的真实来源应该是后端数据库
2. **避免前端被攻击**：前端数据不应被信任
3. **性能优先**：不影响用户交互体验

#### 性能分析

| 方案 | 渲染性能 | 数据一致性 | 复杂度 |
|-----|---------|-----------|-------|
| 每次替换整个数组 | ❌ 差 - 触发所有消息重新渲染 | ✅ 强一致 | 低 |
| 增量更新 | ✅ 好 - 只渲染新增部分 | ⚠️ 需要处理边界 | 中 |
| 乐观更新 + 最终一致 | ✅ 好 | ⚠️ 最终一致 | 中 |

#### 推荐方案：分层策略

**核心思路**：
- **用户消息**：前端乐观更新（立即显示，用户不需要等）
- **助手消息**：前端实时接收 token，后端存储后前端再同步一次
- **数据一致性**：以后端为准，但不影响用户看到的实时响应

#### 实现方案

**场景分析**：

```
发送消息流程：

1. 用户发送 "你好"
   └── 前端：立即显示用户消息（乐观更新）
   └── 后端：存储用户消息
   └── 前端：开始显示助手消息（流式）
   └── 后端：存储助手消息
   └── 前端：收到 done → 可选同步

问题：如果每次都替换整个数组，会导致：
- 用户消息重新渲染（浪费）
- 助手消息重新渲染（已经在流式显示了）
- 页面闪烁/滚动位置跳动
```

**方案 A：增量更新（推荐）**

```typescript
// src/stores/chat.ts
async function sendMessage(content: string) {
  // ... 验证逻辑

  const conversationId = currentConversationId.value!;

  // ✅ 乐观更新：立即显示用户消息
  const tempMessage = {
    id: crypto.randomUUID(),
    conversationId,
    role: 'user' as const,
    content,
    createdAt: new Date().toISOString(),
  };
  messages.value.push(tempMessage);

  isLoading.value = true;

  try {
    await chatApi.send(conversationId, content);

    // ✅ 增量同步：只更新需要同步的部分
    // 方案：后端返回新增的消息 ID，前端只添加这些消息
    // 或者：后端返回最后一条消息，前端只检查并补充

    // 本次实现：发送完成后，不替换整个数组
    // 因为用户消息已经显示了，助手消息也在流式更新
    // 只在后端需要通知前端某些状态变更时（如压缩结果）

  } catch (e) {
    // ❌ 失败时回滚
    const idx = messages.value.findIndex(m => m.id === tempMessage.id);
    if (idx !== -1) {
      messages.value.splice(idx, 1);
    }
  } finally {
    isLoading.value = false;
  }
}
```

**方案 B：静默同步（后台更新）**

```typescript
// src/stores/chat.ts
async function sendMessage(content: string) {
  // ... 乐观更新用户消息

  try {
    await chatApi.send(conversationId, content);

    // ✅ 静默同步：后台请求，不阻塞 UI
    // 使用 requestIdleCallback 或 setTimeout 延迟执行
    setTimeout(async () => {
      try {
        const history = await chatApi.getHistory(conversationId);

        // ✅ 智能合并：只添加缺失的消息
        const currentIds = new Set(messages.value.map(m => m.id));

        history.forEach(msg => {
          if (!currentIds.has(msg.id)) {
            messages.value.push(msg);
          }
        });
      } catch (e) {
        console.warn('[ChatStore] 后台同步失败:', e);
      }
    }, 100); // 延迟 100ms，等待流式更新完成

  } catch (e) {
    // 失败处理
  }
}
```

**方案 C：后端主动推送增量数据（最优但复杂）**

```typescript
// 思路：后端在发送完成后，主动推送增量消息
// 前端只添加收到的新消息

// chat.handler.ts
ipcMain.handle('chat:send', async (_, conversationId, message) => {
  return new Promise((resolve, reject) => {
    core.sendMessage(conversationId, message, {
      onToken: (token) => {
        mainWindow.webContents.send('agent:token', token);
      },
      onDone: () => {
        mainWindow.webContents.send('agent:done', null);

        // ✅ 推送新消息的 ID（而不是全部消息）
        const messages = core.getStorage().getMessages(conversationId);
        const lastMessages = messages.slice(-2); // 只推送最后两条（用户+助手）

        mainWindow.webContents.send('agent:messages-synced', {
          conversationId,
          newMessages: lastMessages,
        });

        resolve({ success: true });
      },
      onError: (error) => {
        mainWindow.webContents.send('agent:error', error);
        reject(new Error(error));
      },
    });
  });
});

// preload.ts
onMessagesSynced: (callback) => {
  ipcRenderer.on('agent:messages-synced', (_, data) => callback(data));
},

// chatStore.ts
setupChatListeners({
  // ...
  onMessagesSynced: (data) => {
    // 只添加新消息
    data.newMessages.forEach(msg => {
      if (!messages.value.find(m => m.id === msg.id)) {
        messages.value.push(msg);
      }
    });
  },
});
```

#### 性能对比

| 方案 | 渲染次数 | 用户感知 | 实现复杂度 |
|-----|---------|---------|-----------|
| 替换整个数组 | N 次（每次消息） | 可能闪烁/卡顿 | 低 |
| 增量更新 | 1 次（完成后） | 无感知 | 中 |
| 静默同步 | 1 次（延迟后） | 无感知 | 中 |
| 主动推送 | 0 次（实时） | 最流畅 | 高 |

#### 推荐：方案 A（增量更新）

**理由**：
1. 简单易实现
2. 用户消息已乐观更新，不需要等后端
3. 助手消息通过流式实时显示
4. 完成后静默同步，确保数据一致性
5. 最小化 Vue 重新渲染次数

## 六、性能优化要点

### 前端优化

1. **乐观更新**：用户消息立即显示，不等待后端
2. **流式渲染**：助手消息通过 onToken 实时追加，不重绘
3. **智能同步**：完成后静默同步，只添加缺失的消息
4. **唯一 Key**：使用消息 ID 作为 Vue 列表的 key

### 后端优化

1. **增量返回**：可只返回最后 N 条消息，而非全部
2. **消息 ID**：返回新消息的 ID 列表，前端按 ID 过滤
3. **异步存储**：消息存储可以是异步的，不阻塞响应
